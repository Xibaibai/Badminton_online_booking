## 一、系统概述

### 1.1 系统目标与范围

**系统目标**：
打造一款面向年轻人的羽毛球运动社交平台，帮助用户快速找到同城球友、便捷预约场地、购买装备，实现"约球不孤单，打球更来电"的产品价值。

**系统范围**：
```
┌─────────────────────────────────────────────────────────────┐
│                    拍档系统业务范围                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  【核心业务】                                                │
│  ├── 用户模块：注册认证、个人资料、运动偏好                  │
│  ├── 活动模块：活动发布、报名、管理、评价                    │
│  ├── 匹配模块：智能推荐、筛选、闪电组队                      │
│  ├── 场地模块：场馆查询、在线预约、订单管理                  │
│  ├── 商城模块：装备展示、购物车、订单、支付                  │
│  └── 社交模块：消息通知、搭子关系、互动评价                  │
│                                                              │
│  【支撑业务】                                                │
│  ├── 实名认证：多级别身份认证                               │
│  ├── 信用体系：用户信用评分与等级                           │
│  ├── 支付系统：活动报名、场地预约、商城购买                  │
│  ├── 消息系统：推送通知、活动群聊                             │
│  └── 数据分析：用户行为、运营数据、业务报表                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**非功能目标**：
- 性能：接口响应时间 < 200ms（P99）
- 可用性：系统可用性 > 99.9%
- 并发：支持10万+同时在线用户
- 安全：满足等保三级要求

### 1.3 用户角色定义

| 角色 | 描述 | 权限范围 |
|------|------|----------|
| 游客 | 未注册用户 | 浏览公开活动（需登录才能报名） |
| 普通用户 | 注册但未实名认证 | 浏览、报名免费活动 |
| 已实名用户 | 完成Lv.2实名认证 | 全部功能权限 |
| 高级用户 | 完成Lv.3高级认证 | 高信用活动优先权 |
| 活动发起人 | 发起活动的用户 | 管理自己发起的活动 |
| 场馆商家 | 入驻平台的场馆 | 管理场馆信息、订单 |
| 管理员 | 平台运营人员 | 内容审核、用户管理、数据报表 |

## 三、技术选型

### 3.1 技术选型原则

**业务场景分析**：
拍档是一款面向年轻人的羽毛球运动社交平台，具有以下业务特点：

| 业务特征 | 分析 | 选型影响 |
|----------|------|----------|
| 高并发 | 活动高峰期（周末、节假日）并发量激增 | 需要弹性扩展能力 |
| 实时性 | 闪电组队、消息通知需要低延迟 | 需要高性能缓存和消息队列 |
| 社交属性 | 用户量大、关系复杂、数据增长快 | 需要可扩展的存储方案 |
| LBS需求 | 附近场地、活动推荐依赖位置服务 | 需要成熟的地图服务 |
| 支付场景 | 活动报名、装备购买需要安全支付 | 需要合规的支付方案 |
| 内容合规 | 实名认证、内容审核 | 需要可靠的认证服务 |

**选型评估维度**：

| 维度 | 说明 | 权重 |
|------|------|------|
| 成本 | 初期投入、运营成本、扩展成本 | 25% |
| 成熟度 | 产品稳定性、功能完备性 | 20% |
| 社区活跃度 | 文档完善度、社区支持 | 15% |
| 团队熟悉度 | 学习成本、技术积累 | 15% |
| 生态整合 | 与其他组件的兼容性 | 10% |
| 安全合规 | 资质认证、数据安全 | 15% |

**选型决策流程**：

### 3.2 前端技术栈选型

#### 3.2.1 移动端App技术方案

| 方案 | 推荐度 | 适用场景 | 优点 | 缺点 | 成本预估 |
|------|--------|----------|------|------|----------|
| **Flutter** | ⭐⭐⭐⭐⭐ | 跨平台开发首选 | 性能接近原生、Dart语言易学、单代码库 | 第三方库相对少 | 中 |
| React Native | ⭐⭐⭐⭐ | 团队熟悉React | JS生态丰富、社区活跃 | 性能略差、频繁更新 | 中 |
| 原生开发(iOS+Android) | ⭐⭐⭐ | 对性能要求极高 | 最佳性能和用户体验 | 双倍开发成本 | 高 |
| uni-app | ⭐⭐⭐ | 快速上线MVP | 一套代码多端、Vue语法 | 性能一般、定制受限 | 低 |

**推荐方案：Flutter**

**选型理由**：
1. 性能优异：编译为原生代码，性能接近原生应用
2. 开发效率高：Dart语言简单易学，热重载提升开发体验
3. 单代码库：同时支持iOS和Android，减少维护成本
4. 生态成熟：Google背书，组件库日趋完善
5. 案例丰富：微信、字节等大厂已在生产环境使用

#### 3.2.2 微信小程序技术方案

| 方案 | 推荐度 | 优点 | 缺点 |
|------|--------|------|------|
| **原生开发** | ⭐⭐⭐⭐⭐ | 性能最优、API最新 | 语法特殊 |
| Taro (React) | ⭐⭐⭐⭐ | 一套代码多端 | 性能损耗 |
| uni-app (Vue) | ⭐⭐⭐⭐ | 一套代码多端 | 定制受限 |

**推荐方案：原生开发**

**选型理由**：
- 小程序性能敏感，原生开发性能最优
- 微信API更新快，原生开发第一时间可用
- 项目规模适中，无需多端复用

#### 3.2.3 Web管理后台技术方案

| 方案 | 推荐度 | 优点 | 缺点 | 适用场景 |
|------|--------|------|------|----------|
| **Vue3 + Vite** | ⭐⭐⭐⭐⭐ | 性能好、生态完善、团队熟悉 | - | 首选方案 |
| React + Ant Design | ⭐⭐⭐⭐ | 组件丰富 | 学习曲线较陡 | 复杂后台 |
| Next.js | ⭐⭐⭐ | SSR支持、SEO友好 | 配置复杂 | 有SEO需求 |

**推荐方案：Vue3 + Vite + Element Plus**

**选型理由**：
- Vue3 Composition API 更适合复杂业务逻辑
- Vite 带来极致的开发体验
- Element Plus 组件丰富，适合快速开发
- 团队技术积累，运维成本低

### 3.3 后端技术栈选型

#### 3.3.1 开发语言选型

| 语言 | 推荐度 | 优点 | 缺点 | 适用场景 |
|------|--------|------|------|----------|
| **Java** | ⭐⭐⭐⭐⭐ | 生态完善、稳定性高、招聘容易 | 语法啰嗦、启动慢 | 企业级应用首选 |
| Go | ⭐⭐⭐⭐ | 性能高、并发强、部署简单 | 泛型弱、生态较新 | 高并发服务 |
| Python | ⭐⭐⭐ | AI能力强、开发快 | 性能较弱 | 算法、脚本 |
| Node.js | ⭐⭐⭐ | 前端友好、JSON处理强 | 单线程、大计算量弱 | 小型服务 |

**推荐方案：Java**

**选型理由**：
- Spring Cloud Alibaba 生态成熟，微服务支持完善

### 5.2 主要数据表结构

**用户表 (users)**
```sql
CREATE TABLE `users` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '用户ID',
  `phone` varchar(20) DEFAULT NULL COMMENT '手机号',
  `nickname` varchar(50) NOT NULL COMMENT '昵称',
  `avatar` varchar(255) DEFAULT NULL COMMENT '头像URL',
  `gender` tinyint DEFAULT 0 COMMENT '性别 0-未知 1-男 2-女',
  `birthday` date DEFAULT NULL COMMENT '生日',
  `city` varchar(50) DEFAULT NULL COMMENT '所在城市',
  `bio` varchar(200) DEFAULT NULL COMMENT '个人简介',
  `level` tinyint DEFAULT 1 COMMENT '羽毛球水平 1-新手 2-入门 3-中级 4-高级 5-高手',
  `auth_level` tinyint DEFAULT 0 COMMENT '认证级别 0-未认证 1-Lv1 2-Lv2 3-Lv3',
  `credit_score` decimal(5,2) DEFAULT 5.00 COMMENT '信用评分',
  `credit_level` tinyint DEFAULT 1 COMMENT '信用等级',
  `status` tinyint DEFAULT 1 COMMENT '状态 1-正常 2-禁用',
  `last_login_at` datetime DEFAULT NULL COMMENT '最后登录时间',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_phone` (`phone`),
  KEY `idx_city` (`city`),
  KEY `idx_level` (`level`),
  KEY `idx_auth_level` (`auth_level`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

**用户认证表 (user_auths)**
```sql
CREATE TABLE `user_auths` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '认证ID',
  `user_id` bigint unsigned NOT NULL COMMENT '用户ID',
  `auth_type` tinyint NOT NULL COMMENT '认证类型 1-手机 2-身份证 3-支付宝 4-微信',
```

```
- page: 页码（默认1）

- sort: 排序字段，如 created_at
```

### 6.1 认证授权机制

**多因素认证**：
```
┌─────────────────────────────────────────────────────────────┐
│                    多因素认证体系                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Level 1: 账号密码                                          │
│  ├── 密码强度要求：8位以上，包含大小写字母和数字             │
│  ├── 密码加密存储：BCrypt                                    │

│                                                              │
│  Level 2: 短信验证码                                        │
│  ├── 验证码有效期：5分钟                                     │
│  ├── 验证码一次性使用                                        │
│  ├── 发送频率限制：60秒/次                                   │
│  └── 每日发送上限：10次                                     │
│                                                              │
│  Level 3: 身份证认证                                        │
│  ├── OCR识别+人工审核                                       │
│  ├── 公安接口实时比对                                        │
│  └── 人脸活体检测                                           │
│                                                              │
│  Level 4: 第三方信用认证                                     │
│  ├── 支付宝/微信实名授权                                    │
│  ├── 芝麻信用/微信支付分验证                                 │
│  └── 社保/公积金验证                                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

```java
// 使用JPA加密字段
@Convert(converter = AesEncryptConverter.class)
@Column(name = "identity_no")
private String identityNo;

// 自定义加密转换器
public class AesEncryptConverter implements AttributeConverter<String, String> {
    
    private static final String KEY = "your-256-bit-key"; // 应从KMS获取
    
    @Override
    public String convertToDatabaseColumn(String attribute) {
        if (attribute == null) return null;
        return AESUtil.encrypt(attribute, KEY);
    }
    
    @Override
    public String convertToEntityAttribute(String dbData) {
        if (dbData == null) return null;
        return AESUtil.decrypt(dbData, KEY);
    }
}
```
